# 仓库指南

## 项目结构与模块组织
当前仓库尚无已提交的源代码或测试目录。后续建议保持简洁的常规结构：

- `src/`：应用或库源代码
- `tests/` 或 `__tests__/`：自动化测试
- `assets/`：静态资源（图片、夹具）
- `scripts/`：开发辅助脚本

如使用非标准结构，请在此处说明。

## 构建、测试与开发命令
目前未提供构建或测试命令。后续可在 `Makefile`、`package.json` 等文件中定义，并在此列出主流程，例如：

- `make build`：编译或打包项目
- `make test`：运行完整测试套件
- `make lint`：静态检查或格式化

## 代码风格与命名约定
在引入格式化或 lint 工具前，请统一缩进（2 或 4 空格择一并保持一致）、合理控制行宽，并统一使用 `kebab-case` 或 `snake_case` 的命名风格。引入工具后（如 `prettier`、`eslint`、`ruff`、`gofmt`），请在此说明并优先使用自动化格式化。

## 测试指南
当前未配置测试框架。引入测试后请注明框架（如 `pytest`、`jest`、`go test`），并采用清晰的命名方式，如 `test_user_creation` 或 `UserCreation.spec.ts`，同时说明覆盖率要求和如何运行单测。

## 提交与 PR 规范
当前工作区无法读取提交历史。建议采用简洁的约定式提交，例如：`feat(auth): add login handler`。PR 应包含：

- 变更内容与动机的清晰说明
- 相关 issue/需求链接（如有）
- 测试证明（命令与结果）
- UI/行为变更的截图或日志

## 安全与配置提示
若引入密钥或配置文件，请提供 `.env.example` 或 `config.example.yml` 并说明必填变量。切勿提交敏感信息。
